			- Example with fixed WebM duration
			  collapsed:: true
				- [GitHub - yusitnikov/fix-webm-duration: navigator.mediaDevices.getUserMedia + MediaRecorder create WEBM files without duration metadata. This library appends missing metadata section right to the file blob.](https://github.com/yusitnikov/fix-webm-duration) / [fix-webm-duration - npm](https://www.npmjs.com/package/fix-webm-duration)
				- ```typescript
				  let mediaRecorder: MediaRecorder;
				  let chunks: Blob[] = [];
				  let startTime: number;
				  
				  const encoderOptions = {
				    audioBitsPerSecond: 256000,
				    videoBitsPerSecond: 7500000,
				    mimeType: "video/webm;codecs=av1"
				  };
				  
				  async function recordOn() {
				    await fixWebm();
				    startTime = Date.now();
				  }
				  
				  async function recordOff() {
				    mediaRecorder.stop();
				  }
				  
				  function fixWebm() {
				    chunks = [];
				  
				    // Replace this with your actual element containing the stream
				    const recordStream = document.getElementById("Your Stream To Be Recorded Here") as HTMLVideoElement;
				    const stream = recordStream.srcObject as MediaStream;
				  
				    mediaRecorder = new MediaRecorder(stream, encoderOptions);
				  
				    mediaRecorder.onstop = function () {
				      const endTime = Date.now();
				      const duration = endTime - startTime;
				  
				      const buggyBlob = new Blob(chunks, { type: 'video/webm' });
				  
				      ysFixWebmDuration(buggyBlob, duration, { logger: false })
				        .then(function (durationFixedBlob) {
				          // Upload to server via XMLHttpRequest if checkbox is checked
				          const uploadToServer = document.getElementById('uploadToServer') as HTMLInputElement;
				          if (uploadToServer.checked) {
				            uploadFileToServer(durationFixedBlob);
				          }
				  
				          console.log('Duration (ms):', duration);
				          download(durationFixedBlob);
				          console.log('Recording ended.');
				        });
				    };
				  
				    mediaRecorder.ondataavailable = function (event) {
				      const data = event.data;
				      if (data && data.size > 0) {
				        chunks.push(data);
				      }
				    };
				  
				    mediaRecorder.start();
				    console.log('Recording started.');
				  }
				  ```

				  collapsed:: true
				  This provides WebSocket support for any application/server. Translates WebSockets traffic into normal socket traffic
					- There's also a JavaScript implementation which is not well maintained.
					- Obtain TAR file docker image for it, then load it into Docker:
						- `docker load -i websockify.tar`
					- Generate a self-signed certificate using `openssl`:
						- `openssl req -new -x509 -days 365 -nodes -out self.pem -keyout self.pem`
						- Copy the certificate section including header and footer `cat self.crt`
					- To run websockify:
						- Docs: [docker run](https://docs.docker.com/reference/cli/docker/container/run/)
						- `docker run -it --rm -p 7777:443 -v ./web-sockify-data:/opt/websockify novnc/websockify 443 172.21.52.55:8888`
							- IP address obtained via `ifconfig` - see `eth0` : `inet` section
							- `8888` = the SSH connection, chosen above
							- `7777` = the secure WebSocket port. This is arbitrarily chosen, but must be unique for each device.
							  docker run: This command starts a new Docker container.
							  `-it`: This flag runs the container in interactive mode, allowing you to interact with the container's terminal.
							  `--rm`: This flag ensures that the container is automatically removed when it exits, which helps to clean up resources.
							  `-p 7777:443`: This maps port 7777 on the host machine to port 443 on the container. This allows you to access the service running on port 443 inside the container from the host machine on port 7777.
							  `-v ./web-sockify-data:/opt/websockify`:
								- `-v` = used to define volume mounts, which allows sharing storage volumes across multiple containers
								  This mounts the ./web-sockify-data directory from the host machine to the /opt/websockify directory inside the container. This is useful for sharing files between the host and the container.
								- When given two arguments (host_path), like `-v /data/mysql:/var/lib/mysql`, this mounts the specified directory on the host inside the container at the specified path. This can also be used to expose host files inside the container; for example, -v /etc/localtime:/etc/localtime would make /etc/localtime on the host available as /etc/localtime inside the container.
								- The first location is on the host system, second in the Docker filesystem
								  novnc/websockify: This specifies the Docker image to use for the container. In this case, it's the novnc/websockify image, which is a WebSocket to TCP proxy/bridge.
								  443 172.21.52.55:8888: These are the arguments passed to the websockify command inside the container. The first argument, 443, is the port on which websockify will listen inside the container. The second argument, 172.21.52.55:8888, specifies the target TCP address and port to which websockify will forward the WebSocket traffic.
							- In summary, this command starts a websockify container that listens on port 443 inside the container (mapped to port 7777 on the host) and forwards WebSocket traffic to the TCP address 172.21.52.55:8888. The container is run in interactive mode, will be removed when it exits, and has a volume mounted to share files.

				- WebTransport
				  id:: 67c0e9ad-6701-45a0-883c-6b6674c15c9f
				  collapsed:: true
					- Overview
					  collapsed:: true
						- A technology that enables bidirectional, full-duplex communication between client and server over a persistent, single-socket connection. This allows for multiple streams, unidirecctional streams, out-of-order delivery, low-latency, realtime updates, and reliable and unreliable transport.
						- A pluggable protocol for client-server communication, built on top of HTTP/2, HTTP/3, and QUIC. It is designed to replace WebSockets going ‘QUIC-native.’
						- Successor to WebRTC and WebSockets for real-time web communication
							- WebSockets involves one server having bidirectional communication to multiple browsers.
							- WebRTC involves one server having bidirectional communication to multiple clients, and the clients also have bidirectional communication with each other.
								- Communication happens between browsers without requiring the direct involvement of the server. However, the server needs to facilitate sharing each other’s IP address at the beginning. Still, it’s faster than communicating through a server.
					- In-depth
						- Already has APIs for sending data both reliably and unreliably.
						  collapsed:: true
							- In a reliable data transfer, the sender is notified of the success or failure of the data transmission, and failed transmissions are usually resent until they succeed, after which the next data packet is sent. In unreliable transfer, there’s no confirmation of transmission success, and packets that aren’t received simply don’t get delivered. Unreliable transfer is often used for things like streaming videos, where speed is a concern, and minor data loss, such as a few frames of video, is acceptable. Because WebTransport uses both of these methods, there are many use cases for it, such as bidirectional data streaming for multiplayer gaming, interactive live streams, and data transfer for sensors and internet of things devices.
						- With WebTransport, you can also use features like promises and the await keyword for asynchronous functions. The API runs in Web Workers, too, enabling multithreading.
						- Can send multiple types of data over the same connection. That means you can send and receive video information unreliably while sending text or file data reliably.
						- WebTransport is on top of QUIC
						  collapsed:: true
							- WebTransport is an interface that can talk to HTTP/2 based, HTTP/3 based, and QUIC based protocols. So, it has the advantage that HTTP and non-HTTP traffic can share the same network port.
							- Besides, since QUIC operates over UDP, each stream is independent. Switching to UDP is an advantage to reduce the impact of TCP head of line blocking. Therefore any lost packet only halts the stream that it belongs to, while the other streams can go on.
						- Supports multiple protocols
						  collapsed:: true
							- supports unidirectional streams (indefinitely long streams of bytes in one direction), bidirectional streams (full-duplex streams), and datagrams (small/out-of-order/unreliable messages)
						- It's intended to replace or supplement existing technologies like Long Polling, WebSockets, XMLHttpRequest, and Fetch.
					- Differences to WebSockets
						- While WebSockets creates a single stream per connection, WebTransport allows you to create multiple streams over a single connection. It avoids the head-of-line blocking delays that WebSockets suffers from, and is less resource intensive when creating connections. While WebSockets starts as a HTTP/1.1 protocol, WebTransport works on top of several different protocols, including some that WebSockets don't support.
						  collapsed:: true
							- It works with HTTP/3, the upcoming version of the transport protocol used by the World Wide Web. HTTP/3 uses the QUIC protocol for transport layer data exchange, which has several advantages. QUIC can prevent head-of-line blocking delays, improving network performance in many situations. This is a limitation of WebSockets.
						- WebTransport uses QUIC protocol, WebSockets uses TCP
						  collapsed:: true
							- As WebTransport uses the QUIC protocol, these connections are less resource intensive to open and close
					- Examples
					  collapsed:: true
						- Here’s an example of an async function from the WebTransport Working Group’s [documentation](https://www.w3.org/TR/webtransport/#example-sending-stream).

							  async function sendData(url, data) {
							  const wt = new WebTransport(url);
							  const writable = await wt.createUnidirectionalStream();
							  const writer = writable.getWriter();
							  await writer.write(data);
							  await writer.close();
							  }

					- Implementations
						- [webtransport](https://www.w3.org/TR/webtransport/): This is a JavaScript library developed by the W3C WebTransport Working Group. It provides a high-level API for establishing WebTransport connections and sending/receiving data over the connection.
						- [quiche](https://github.com/cloudflare/quiche): This is a Rust library developed by CloudFlare. It is a general-purpose QUIC library that can also implement WebTransport. Quiche provides low-level control over the QUIC protocol and allows for fine-grained customization.
						- [aioquic](https://pypi.org/project/aioquic/): This Python library provides a high-level API for implementing QUIC and WebTransport. It is built on the quiche library and easily integrates WebTransport into your Python applications.
						- [libwebsockets](https://libwebsockets.org/): This C library provides a lightweight and efficient way to implement WebTransport. It supports client- and server-side implementations and offers a flexible API for sending and receiving data over WebTransport connections. Alternatively, you can use other server implementations like Warp. Choose the server implementation that best suits your needs and follow the documentation to set it up.
